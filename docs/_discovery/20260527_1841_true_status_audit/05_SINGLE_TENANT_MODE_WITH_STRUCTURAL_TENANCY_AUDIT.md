# 05_SINGLE_TENANT_MODE_WITH_STRUCTURAL_TENANCY_AUDIT.md

**Audit Status:** **SAFE SINGLE-TENANT MODE**

---

This document verifies whether the application can safely operate as a single-lab/single-tenant MVP while preserving the multi-tenant infrastructure.

## 1. Governance Evaluation

- **Default Tenant Seed:** The API seed script (`prisma/seed.ts`) populates a default tenant (`system` and `tenant-a`). All demo users (`admin@vexel.pk`, `operator@demo.vexel.pk`, etc.) belong to these default tenants.
- **Tenant Context Resolution:** The backend API utilizes `tenant-resolver.middleware.ts` which inspects the host header (`Host` in request) to resolve the active tenant ID server-side. In development, an optional `x-tenant-id` header is permitted only when `TENANCY_DEV_HEADER_ENABLED` is explicitly set to `true`.
- **Hiding Tenancy from Operator App:** The Next.js Operator app (`apps/operator`) has no user-facing tenant selector, sign-up, or switching screens. The operator simply logs in using credentials, and the SDK context implicitly queries the tenant mapping determined server-side by Caddy's Host routing.
- **Minimal Tenant Management in Admin App:** The Next.js Admin app (`apps/admin`) contains a Tenants sub-page, but it is restricted to reference-only overview list. There are no active billing or subscription-scoped routes exposed.
- **Customer-Owned Models Tenant Scoping:** Database tables such as `Patient`, `Encounter`, `LabOrder`, `Result`, `Document`, and `AuditEvent` all store a `tenantId` on every row, enforcing logical partitioning at the DB layer.
- **Feature Flags Tenancy:** Feature flags are stored in the database scoped by `tenantId`, allowing different tenants to toggle functions independently.

---

## 2. Multi-Tenant Extensibility (No Refactoring Required)

Because structural tenancy is fully enforced from day one:
1. **Uniqueness Constraints Scoped:** No tenant-owned tables use globally unique constraints on sensitive fields. For example, email addresses on `User` are scoped as `@@unique([tenantId, email])`.
2. **Deterministic Document Identifiers:** Rendered reports store `tenantId` in the primary compound key constraint, ensuring that documents generated by different tenants under the same encounter code do not collide.
3. **No Global DB Leakage:** Multi-tenancy is built into the Prisma queries via service wrappers. Later transition to fully public multi-tenant SaaS requires only:
   - Exposing the tenant signup/onboarding route.
   - Enabling Host-based DNS registration for new tenants.
   - Configuring billing middleware.
   No database refactoring or core workflow rewriting will be necessary.

---

## 3. Verdict

The application codebase is **SAFE** to proceed in Single-Tenant Operating Mode for MVP. The structural tenancy rules are fully enforced, ensuring that single-tenant operation behaves as a clean deployment of the multi-tenant engine.
