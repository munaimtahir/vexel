# SDK Generation and Usage Audit

Primary evidence:
- SDK file inventory: `logs/phase5_sdk_files.txt`
- SDK import references: `logs/phase5_sdk_imports_all.txt`
- OpenAPI-derived expected types (evidence-only): `contracts/openapi/sdk_expected_api.d.ts`
- SDK generated types in repo: `packages/sdk/src/generated/api.d.ts`
- Diff proving equality: `logs/phase5_sdk_api_dts_diff.txt` and exit code `logs/phase5_sdk_api_dts_diff.exitcode.txt`

## Findings (from scratch)

1. SDK package exists and contains a generated types surface
- Package: `packages/sdk` (Evidence: `logs/phase5_sdk_files.txt`)
- Generated types file: `packages/sdk/src/generated/api.d.ts` (Evidence: `logs/phase5_sdk_files.txt`)

2. SDK appears to be used by both Admin and Operator via wrappers
- Admin imports `@vexel/sdk` in:
  - `apps/admin/src/lib/api-client.ts`
  - `apps/admin/src/lib/auth.ts`
- Operator imports `@vexel/sdk` in:
  - `apps/operator/src/lib/api-client.ts`
  - `apps/operator/src/lib/auth.ts`
(Evidence: `logs/phase5_sdk_imports_all.txt`)

3. Contract → SDK generated types freshness (static proof)

Without modifying repo SDK output, this audit generated an expected `api.d.ts` directly from the canonical OpenAPI using `openapi-typescript` into the evidence folder and diffed it against the repo’s `packages/sdk/src/generated/api.d.ts`.

Result:
- Diff exit code `0` (no differences). (Evidence: `logs/phase5_sdk_api_dts_diff.exitcode.txt`)

Implication:
- The current `packages/sdk/src/generated/api.d.ts` matches `packages/contracts/openapi.yaml` as generated by `openapi-typescript` (tool version recorded in Phase 4).
